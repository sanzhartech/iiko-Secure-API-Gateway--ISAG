# Журнал отладки — Стабильность аутентификации и тестирования

## Баг 1: settings_init_missing_files
- **Проблема**: `FileNotFoundError` при инициализации Pydantic-класса `Settings` в тестах.
- **Причина**: Метод `Settings.model_post_init` считывал RSA-ключи с диска сразу при запуске. Тесты использовали `MagicMock`, который не проходил валидацию Pydantic или не имел реальных дескрипторов файлов.
- **Решение**: Обновлен файл `conftest.py` с использованием фикстуры `tmp_path_factory` для генерации реальных временных файлов `.pem` и `.json` перед инициализацией объекта `Settings`.
- **Статус**: ✅ Решено

## Баг 2: redundant_patch_conflicts
- **Проблема**: Тесты падали с кодами 404 или 422, даже когда токены были валидными.
- **Причина**: Сложные вложенные блоки `with patch(...)` в `test_jwt.py` перезаписывали зависимости `async_client`, что приводило к несогласованному состоянию конфигурации.
- **Решение**: Удалены все ручные патчи `get_settings`. Настройка зависимостей стандартизирована через `app.dependency_overrides` в фикстуре `async_client`.
- **Статус**: ✅ Решено

## Баг 3: mock_context_manager_typeerror
- **Проблема**: `TypeError: 'builtins.coroutine' object does not support the asynchronous context manager protocol`.
- **Причина**: Функция `proxy_request_stream` является контекстным менеджером потоковой передачи (`@asynccontextmanager`). Мокирование через стандартный `AsyncMock` вызывало ошибку, когда приложение пыталось использовать ее в конструкции `async with`.
- **Решение**: Моки в тестах заменены на реальные вспомогательные функции с декоратором `@asynccontextmanager` для корректной имитации потоковой передачи данных.
- **Статус**: ✅ Решено

## Баг 4: missing_sub_claim_keyerror
- **Проблема**: Ошибка 500 Internal Server Error, когда в токене JWT отсутствовал клайм `sub`.
- **Причина**: Метод `JWTValidator.validate` обращался к `payload["sub"]` напрямую без предварительной проверки существования, нарушая принцип «Fail-Closed» (блокировка при сбое).
- **Решение**: Добавлена явная проверка наличия клаймов `sub` и `iss`, с вызовом метода `_reject()` для возврата корректного ответа 401 вместо падения сервера.
- **Статус**: ✅ Решено

## Баг 5: rate_limit_limiter_divergence
- **Проблема**: Тест `test_auth_endpoint_rate_limit_behavior` не приводил к ошибке 429 при превышении лимита.
- **Причина**: В `main.py` создавался новый экземпляр `Limiter` для состояния приложения, в то время как декоратор `@limiter.limit` на маршруте ссылался на исходный синглтон на уровне модуля. Они использовали разные хранилища в памяти.
- **Решение**: Файл `main.py` перенастроен на работу с глобальным синглтоном `limiter`, а его лимиты по умолчанию обновляются во время выполнения.
- **Статус**: ✅ Решено

## Баг 6: test_fixture_roles_defaulting
- **Проблема**: Тест `test_no_roles_denied` возвращал код 200 вместо ожидаемого 403.
- **Причина**: В фикстуре `make_token` использовалась конструкция `roles or ["operator"]`. Передача пустого списка `roles=[]` оценивалась в Python как `False`, из-за чего токену автоматически назначалась роль `operator`.
- **Решение**: Логика изменена на `roles if roles is not None else ["operator"]`.
- **Статус**: ✅ Решено

- **Функционал**: Реализован эндпоинт `/auth/refresh` с полным разделением типов токенов.
- **Изменения**:
