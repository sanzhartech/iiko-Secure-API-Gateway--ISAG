# Техническая документация проекта iiko Secure API Gateway (ISAG)

## 1. Математическое и криптографическое обоснование алгоритма JWT RS256

В системе ISAG для аутентификации и авторизации используется стандарт JSON Web Token (JWT) с алгоритмом цифровой подписи **RS256** (RSA Signature with SHA-256). Выбор данного алгоритма обусловлен архитектурными требованиями безопасности и принципом минимальных привилегий.

### Сравнение HS256 и RS256
В отличие от симметричного алгоритма HS256, использующего общий секретный ключ (shared secret), RS256 основан на асимметричном шифровании с парой ключей:
- **Закрытый ключ (Private Key):** Используется сервером аутентификации (ISAG Auth) для генерации подписи. Доступ к нему ограничен только процессом выпуска токенов.
- **Открытый ключ (Public Key):** Используется прокси-сервером или внешними потребителями для проверки подлинности токена.

### Математическое обоснование
Цифровая подпись RS256 базируется на криптосистеме RSA, безопасность которой опирается на сложность задачи факторизации произведения двух больших простых чисел. Процесс формирования подписи $S$ для сообщения $M$ описывается уравнением:
$$S = (H(M))^d \pmod{n}$$
где $H(M)$ — хэш-сумма сообщения (SHA-256), $d$ — закрытая экспонента, $n$ — модуль.

Проверка подписи осуществляется через открытую экспоненту $e$:
$$H(M) = S^e \pmod{n}$$

**Преимущества для ISAG:**
1. **Отсутствие компрометации секрета:** Даже если открытый ключ прокси-сервера будет перехвачен, злоумышленник не сможет генерировать валидные токены.
2. **Масштабируемость:** Открытые ключи могут безопасно распространяться среди множества микросервисов для децентрализованной проверки JWT.
3. **Ротация ключей (KID):** Система поддерживает идентификаторы ключей (Key ID), что позволяет проводить плановую ротацию без остановки сервиса.

## 2. Алгоритм ограничения частоты запросов (Rate Limiting)

В ISAG реализован алгоритм **Fixed Window (Фиксированное окно)**, интегрированный с распределенным хранилищем Redis для обеспечения согласованности данных между несколькими экземплярами шлюза.

### Принцип работы
Алгоритм разделяет временную шкалу на фиксированные интервалы (например, 1 минута). Для каждого идентификатора (IP-адрес или User ID) создается счетчик в Redis с ключом, включающим временную метку окна.

**Логика выполнения:**
1. При поступлении запроса формируется ключ: `isag:limiter:<key>:<timestamp>`.
2. Команда `INCR` атомарно увеличивает значение счетчика.
3. Если значение превышает лимит $L$, запрос отклоняется с кодом 429.
4. Устанавливается `TTL` (Time To Live) для ключа, равный длительности окна, для автоматической очистки памяти.

### Обоснование выбора
Выбор Fixed Window обусловлен его высокой производительностью и низким потреблением ресурсов Redis. Несмотря на проблему "всплеска на границе окон" (boundary burst), алгоритм эффективно защищает upstream-сервисы iiko от DoS-атак и злоупотребления API в условиях высокой нагрузки.

## 3. Структура базы данных для аудита (Audit Logs)

Для обеспечения подотчетности и возможности расследования инцидентов (Incident Response), система ведет детальный журнал действий администраторов. Данные хранятся в PostgreSQL в таблице `admin_audit_logs`.

### Описание схемы
| Поле | Тип данных | Описание | Индекс |
| :--- | :--- | :--- | :--- |
| `id` | `UUID` | Первичный ключ, уникальный идентификатор записи. | PK |
| `timestamp` | `DateTime` | Время выполнения действия (UTC). | Да |
| `admin_id` | `String(128)` | Идентификатор администратора (subject из JWT). | Да |
| `action` | `String(64)` | Тип выполненной операции (например, `CLIENT_CREATED`). | Да |
| `target_id` | `String(128)` | Идентификатор объекта, над которым произведено действие. | Нет |
| `ip_address` | `String(64)` | IP-адрес, с которого был выполнен запрос. | Нет |

### Назначение
Индексация по полям `timestamp`, `admin_id` и `action` позволяет оперативно строить отчеты о подозрительной активности и фильтровать логи по конкретным операторам в панели управления.

## 4. Настройка мониторинга и Grafana

Система ISAG экспортирует метрики в формате Prometheus. Основным показателем является счетчик `isag_requests_total`, который позволяет отслеживать общую нагрузку и качество обслуживания.

### Инструкция по настройке Grafana
1. **Подключение источника данных:** Добавьте Prometheus (`http://prometheus:9090`) в разделе Data Sources.
2. **Создание панели:** Создайте новый Dashboard и добавьте Graph/TimeSeries панель.

### PromQL запросы для метрик
Для визуализации используйте следующие запросы:

**А) Количество запросов в секунду (RPS):**
```promql
rate(isag_requests_total[1m])
```
*Этот запрос вычисляет среднюю интенсивность входящего трафика за последнюю минуту.*

**Б) Соотношение успешных и ошибочных запросов:**
```promql
sum by (status_code) (rate(isag_requests_total[5m]))
```
*Позволяет группировать трафик по HTTP кодам ответов (2xx, 4xx, 5xx).*

**В) Процент заблокированных запросов (Rate Limit hits):**
```promql
sum(rate(isag_blocked_requests_total[1m])) / sum(rate(isag_requests_total[1m])) * 100
```
*Метрика показывает долю трафика, отсекаемого системой защиты.*

### Настройка Dashboard Provisioning
В проекте предусмотрена автоматическая настройка через `provisioning/dashboards`, что позволяет развертывать готовые панели мониторинга при запуске `docker-compose up`.
