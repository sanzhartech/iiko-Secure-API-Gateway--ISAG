# Техническая и математическая документация iiko Secure API Gateway (ISAG)

Данный документ содержит математическое, криптографическое и алгоритмическое обоснование механизмов безопасности шлюза ISAG, подготовленное для защиты дипломного проекта.

---

## 1. Математическое и криптографическое обоснование алгоритма JWT RS256

В системе ISAG аутентификация и авторизация строятся на базе стандарта **JSON Web Token (JWT)** с использованием алгоритма асимметричной цифровой подписи **RS256** (RSA Signature with SHA-256).

### Сравнение симметричного (HS256) и асимметричного (RS256) подходов
*   **HS256 (HMAC with SHA-256)**: Использует один и тот же секретный ключ (shared secret) для подписи и для ее проверки. При компрометации прокси-сервера злоумышленник получает возможность выпускать валидные токены.
*   **RS256 (RSA Signature with SHA-256)**: Использует пару ключей — закрытый (Private Key) для подписи и открытый (Public Key) для проверки. Закрытый ключ хранится исключительно в доверенной зоне шлюза, а открытый может безопасно распространяться.

### Математическая модель подписи RSA
Пусть $p$ и $q$ — два больших случайных простых числа. Вычисляется модуль $n = p \cdot q$.
Вычисляется функция Эйлера:
$$\varphi(n) = (p-1) \cdot (q-1)$$
Выбирается открытая экспонента $e$, взаимно простая с $\varphi(n)$ (обычно $e = 65537$). Вычисляется закрытая экспонента $d$, удовлетворяющая сравнению:
$$d \cdot e \equiv 1 \pmod{\varphi(n)}$$

Открытый ключ состоит из пары $(e, n)$, закрытый ключ — из пары $(d, n)$.

**Процесс формирования подписи ($S$)**:
Для сообщения (заголовка и полезной нагрузки JWT) $M$ вычисляется криптографический хэш SHA-256:
$$h = H(M)$$
Затем формируется подпись с использованием закрытой экспоненты $d$:
$$S = h^d \pmod{n}$$

**Процесс верификации подписи**:
Получатель токена расшифровывает подпись $S$ с помощью открытой экспоненты $e$ и сверяет с вычисленным хэшем сообщения:
$$h' = S^e \pmod{n}$$
Подпись считается валидной тогда и только тогда, когда $h' = H(M)$.

Безопасность схемы основана на вычислительной сложности задачи факторизации модуля $n$ (разложения на простые множители $p$ и $q$).

## 2. Алгоритм ограничения частоты запросов (Rate Limiting)

Для защиты от DDoS-атак и злоупотребления API в ISAG реализован алгоритм **Fixed Window (Фиксированное окно)**, использующий Redis как распределенное хранилище счетчиков.

### Принцип работы
Временная шкала разбивается на равные интервалы (окна) длительностью $W$ секунд (например, $W = 60$).
Каждому клиенту ставится в соответствие лимит $L$ запросов за окно.

**Алгоритм обработки запроса**:
1.  При получении запроса определяется уникальный ключ ограничения $K$:
    *   Если клиент авторизован: $K = \text{userId}$ (взят из проверенного клейма `sub` токена).
    *   Если клиент не авторизован: $K = \text{clientIP}$.
2.  Вычисляется текущее временное окно:
    $$T = \lfloor \text{currentTime} / W \rfloor$$
3.  Формируется ключ счетчика в Redis:
    $$\text{Key} = \text{"isag:limiter:"} + K + \text{":"} + T$$
4.  Выполняется атомарная команда `INCR` счетчика:
    $$\text{currentCount} = \text{redis.INCR(Key)}$$
5.  Если $\text{currentCount} == 1$, устанавливается время жизни ключа (TTL):
    $$\text{redis.EXPIRE(Key, } W)$$
6.  Если $\text{currentCount} > L$, шлюз возвращает ошибку `HTTP 429 Too Many Requests`. Иначе запрос пропускается к проксируемому сервису.

## 3. Алгоритм защиты от атак повторения (JTI Replay Protection)

Для предотвращения повторного использования перехваченных JWT (особенно долгоживущих refresh-токенов) шлюз отслеживает уникальные идентификаторы токенов (`jti`).

### Математическая и алгоритмическая модель
Каждому токену при создании присваивается глобальный уникальный идентификатор $\text{jti}$ (UUIDv4).
При валидации токена проверяется, не выполнялся ли ранее запрос с данным $\text{jti}$.

С целью минимизации ложных срабатываний из-за сетевых повторов или параллельных запросов от клиента, реализован механизм **двухэтапного скользящего окна допуска (Grace Window)**.

**Схема проверки JTI в Redis**:
1.  Формируются ключи:
    $$\text{Key}_{\text{jti}} = \text{"isag:jti:"} + \text{jti}$$
    $$\text{Key}_{\text{grace}} = \text{"isag:jti:grace:"} + \text{jti}$$
2.  Выполняется попытка записи времени первого использования:
    $$\text{oldValue} = \text{redis.SET(Key}_{\text{jti}}\text{, currentTime, EX=tokenTTL + buffer, NX=True, GET=True)}$$
3.  Если $\text{oldValue} == \text{None}$ (ключ записан впервые) $\to$ **Пропуск** (первое использование токена).
4.  Если ключ существовал, вычисляется прошедшее время:
    $$\Delta t = \text{currentTime} - \text{oldValue}$$
5.  Если $\Delta t > \text{GracePeriod}$ (где $\text{GracePeriod} = 2$ сек) $\to$ **Блокировка** (попытка атаки повторения).
6.  Если $\Delta t \le \text{GracePeriod}$ (запрос попал в окно сетевого допуска):
    Выполняется инкремент счетчика параллельных обращений:
    $$\text{graceCount} = \text{redis.INCR(Key}_{\text{grace}}\text{)}$$
    Если $\text{graceCount} == 1$, устанавливается TTL для счетчика:
    $$\text{redis.EXPIRE(Key}_{\text{grace}}\text{, GracePeriod} + 1)$$
    
    *   Если $\text{graceCount} == 1 \to$ **Пропуск** (допускается ровно одно сетевое дублирование).
    *   Если $\text{graceCount} > 1 \to$ **Блокировка** (превышено число допустимых повторных пакетов в окне).

## 4. Архитектура базы данных

Реестр партнеров (клиентов) и журналы действий хранятся в реляционной базе данных.

### Схема таблицы клиентов (`gateway_clients`)
Таблица хранит учетные записи приложений, использующих шлюз.

| Поле | Тип данных | Описание |
| :--- | :--- | :--- |
| `id` | `UUID` (PK) | Уникальный первичный ключ записи. |
| `client_id` | `VARCHAR(128)` (Unique, Index) | Идентификатор клиента. |
| `hashed_secret` | `VARCHAR(512)` | Хэш секрета клиента (алгоритм Bcrypt). |
| `roles` | `JSON` | Роли клиента (например, `["operator"]`). |
| `scopes` | `JSON` | Права доступа/области видимости (например, `["orders:read"]`). |
| `rate_limit` | `INTEGER` | Персональный лимит частоты запросов. |
| `is_active` | `BOOLEAN` | Флаг активности аккаунта (блокировка при `False`). |
| `last_used_at` | `TIMESTAMP` | Время последнего успешного обращения к шлюзу. |

### Схема таблицы аудита действий администраторов (`admin_audit_logs`)
Используется для расследования инцидентов ИБ.

| Поле | Тип данных | Описание |
| :--- | :--- | :--- |
| `id` | `UUID` (PK) | Уникальный идентификатор записи лога. |
| `timestamp` | `TIMESTAMP WITH TIME ZONE` (Index) | Точное время фиксации действия (UTC). |
| `admin_id` | `VARCHAR(128)` (Index) | Идентификатор администратора (sub). |
| `action` | `VARCHAR(64)` (Index) | Тип действия (например, `CLIENT_CREATED`, `KILL_SWITCH_ACTIVE`). |
| `target_id` | `VARCHAR(128)` | Идентификатор изменяемого объекта. |
| `ip_address` | `VARCHAR(64)` | Сетевой адрес администратора. |

### Схема таблицы статистики запросов (`gateway_request_logs`)
Используется для построения аналитических графиков в панели управления.

| Поле | Тип данных | Описание |
| :--- | :--- | :--- |
| `id` | `UUID` (PK) | Идентификатор записи. |
| `timestamp` | `TIMESTAMP WITH TIME ZONE` (Index) | Время прохождения запроса. |
| `client_id` | `VARCHAR(128)` (Index) | Идентификатор клиента. |
| `status_code` | `INTEGER` | HTTP-код ответа. |
| `latency_ms` | `FLOAT` | Время обработки запроса шлюзом в миллисекундах. |

## 5. Метрики Prometheus и мониторинг Grafana

Шлюз экспортирует метрики на эндпоинт `/metrics` в формате OpenMetrics для сбора сервером Prometheus.

### Описание основных метрик
1.  `isag_requests_total{method, path, status_code}`: Счетчик всех обработанных запросов. Поле `path` нормализовано (переменные заменены на `{path}`).
2.  `isag_blocked_requests_total{reason}`: Счетчик запросов, отклоненных системой безопасности (причины: `rate_limit`, `invalid_token`, `expired_token`, `replay_attack`, `kill_switch`).
3.  `isag_request_latency_seconds_bucket{le}`: Гистограмма времени обработки запросов прокси-сервером.

### PromQL-запросы для мониторинга в Grafana
*   **Интенсивность запросов (RPS)**:
    ```promql
    sum(rate(isag_requests_total[1m]))
    ```
*   **Уровень ошибок (Error Rate - % ответов 5xx)**:
    ```promql
    sum(rate(isag_requests_total{status_code=~"5.."}[5m])) / sum(rate(isag_requests_total[5m])) * 100
    ```
*   **Эффективность защиты шлюза (% заблокированных запросов от общего числа)**:
    ```promql
    sum(rate(isag_blocked_requests_total[1m])) / (sum(rate(isag_requests_total[1m])) + sum(rate(isag_blocked_requests_total[1m]))) * 100
    ```
*   **Процентиль времени ответа (95-й процентиль задержки)**:
    ```promql
    histogram_quantile(0.95, sum(rate(isag_request_latency_seconds_bucket[5m])) by (le))
    ```
